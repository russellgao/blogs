## 镜像加速
可以在 `/etc/docker/daemon.json` 中配置加速地址 ,如：
```shell script
{
  "registry-mirrors": [
    "https://hub-mirror.c.163.com",
    "https://mirror.baidubce.com"
  ]
}
```
重启 docker 服务 

>* 这一步是可选的，从 dockerhub 中 pull 镜像比较慢，用于加速

### 启动 portal
```shell script
version=1.8.1
#ddocker pull apolloconfig/apollo-portal:${version}
docker rm -f apollo-portal
docker run -p 8070:8070 \
    -e SPRING_DATASOURCE_URL="jdbc:mysql://xxxxxxxxx:3306/ApolloPortalDB?characterEncoding=utf8" \
    -e SPRING_DATASOURCE_USERNAME=xxxxxxxxx -e SPRING_DATASOURCE_PASSWORD=xxxxxxxxx \
    -e APOLLO_PORTAL_ENVS=dev \
    -e DEV_META=http://xxxxxxxxx:8080 \
    -e SPRING.LDAP.BASE="xxxxxxxxx" \
    -e SPRING.LDAP.USERNAME="xxxxxxxxx" \
    -e SPRING.LDAP.PASSWORD="xxxxxxxxx" \
    -e SPRING.LDAP.SEARCHFILTER="xxxxxxxxx" \
    -e SPRING.LDAP.URLS="ldap://xxxxxxxxx:389" \
    -e LDAP.MAPPING.OBJECTCLASS=user \
    -e LDAP.MAPPING.LOGINID=sAMAccountName \
    -e LDAP.MAPPING.USERDISPLAYNAME=cn \
    -e LDAP.MAPPING.EMAIL=mail \
    -e SPRING.PROFILES.ACTIVE=github,ldap \
    -d -v /home/apollo/portal/logs:/opt/logs --name apollo-portal apolloconfig/apollo-portal:${version}
```

这里可以启动多个服务，通过服务均衡的方式发布服务，根据实际情况部署即可。

特别说明: 

- 这里启动 ldap 认证，如果不需要 ldap 认证则需要去掉 ldap 相关的环境变量（主要是 `SPRING.PROFILES.ACTIVE`）
- Apollo 配置文件中的配置都可以通过环境的方式生效而不用进到容器中修改
- 如果要采用其他认证方式参考 [用户登录功能](https://ctripcorp.github.io/apollo/#/zh/development/portal-how-to-implement-user-login-function)
- 这里只配置了一个环境，如果要支持多环境需要修改 `APOLLO_PORTAL_ENVS` 和 `XXX_META`
- 启动了 LDAP 之后，默认所有用户到不是超级管理员，设置用户为超级管理员的方法，在 `apolloportaldb.serverconfig` 中配置即可，如：
    ```sql
    -- 多个user 用 逗号分隔
    INSERT INTO `apolloportaldb`.`serverconfig`(`Key`, `Value`, `Comment`) VALUES ('superAdmin', 'user1,user2,...', 'Portal超级管理员');
    ```

到这里差不多就已经部署完成了，可以通过 LDAP 愉快的登录了。

## 架构设计
参考: [https://ctripcorp.github.io/apollo/#/zh/design/apollo-design](https://ctripcorp.github.io/apollo/#/zh/design/apollo-design)

- Config Service提供配置的读取、推送等功能，服务对象是Apollo客户端
- Admin Service提供配置的修改、发布等功能，服务对象是Apollo Portal（管理界面）
- Config Service和Admin Service都是多实例、无状态部署，所以需要将自己注册到Eureka中并保持心跳
- 在Eureka之上我们架了一层Meta Server用于封装Eureka的服务发现接口
- Client通过域名访问Meta Server获取Config Service服务列表（IP+Port），而后直接通过IP+Port访问服务，同时在Client侧会做load balance、错误重试
- Portal通过域名访问Meta Server获取Admin Service服务列表（IP+Port），而后直接通过IP+Port访问服务，同时在Portal侧会做load balance、错误重试
- 为了简化部署，我们实际上会把Config Service、Eureka和Meta Server三个逻辑角色部署在同一个JVM进程中
